# 快速上手

## 1. 烧录ESP32驱动板固件

正常工作模式下（树莓派/Mac直接控制舵机）：

- 用Arduino IDE打开`firmware/SerialBridge/SerialBridge.ino`并烧录。
- 这个固件让板子变成一个透明的USB↔舵机总线字节转发器——没有WiFi，没有自己的逻辑。主机直接跟舵机说SCServo协议，板子只是个透传的桥。

如果你需要检查接线、确认某个舵机有没有响应、或者改一个舵机的总线ID，临时改烧`firmware/ServoJog/ServoJog.ino`：

- 它会自建一个WiFi热点`ArmServoCtrl`（密码`12345678`）。手机连上这个热点，浏览器打开`192.168.4.1`。
- 长按按钮可以点动各个关节；有OLED状态显示，还有一个"Set ID"输入框可以永久修改舵机的总线ID。
- 平时正常工作不要一直烧着这个固件——它和`SerialBridge`都要独占舵机的串口，同一时间只能用一个。测试完记得改烧回`SerialBridge.ino`。

## 2. 配置主机环境（树莓派或开发机）

```bash
sudo apt install -y python3-picamera2   # 相机库——用apt装，不要用pip（仅树莓派需要）
cd new
pip install -r requirements.txt
```

确认舵机SDK暴露出来的接口跟`arm_hardware.py`期望的一致（不同来源的包命名可能不一样）：

```bash
python3 -c "import scservo_sdk as s; print([n for n in dir(s) if not n.startswith('_')])"
```
找一下有没有`PortHandler`和`PacketHandler`。如果你装的版本接口不一样，需要改的就是`arm_hardware.py`这一个文件——其他地方都不依赖具体的接口名字。

你还需要几张打印出来的AprilTag，`tag36h11`家族（可以从AprilRobotics的`apriltag-imgs`仓库拿，或者随便一个tag36h11生成器）：4张贴在200x150mm工作纸面的四个角，1张装在末端执行器上。

## 3. 为你的机械臂配置calib.json

把`calib.example.json`复制一份改名成`calib.json`，编辑其中的`hardware.servo_port`（如果你的舵机总线ID跟默认值不一样，也改一下`hardware.joint_ids`），让它匹配你实际的设备。其余的内容——运动学参数、运动调优参数——都有合理的默认值，等你做完标定（第6步）会自动更新。

如果你的机械臂是servo2的机身固定在L1末端，但转动轴实际上偏离L1连线一侧（总线舵机很常见——机身有实际宽度，输出轴不可能正好落在连杆中心线上），现在就把这个偏移量（毫米）填进`kinematics.elbow_offset_mm`——直接物理测量两个转轴之间的中心距（卡尺或CAD图纸），不要指望第6步的视觉标定能测出这个值：这个值从数学上就没法只靠末端位置数据反推出来，不管你采集多少数据都一样（想看完整推导可以看README.md的设计说明）。如果两者本来就共线，保持`0.0`不用改。

## 4. 设置机械安全限位（务必在做任何点动/扫描之前完成）

**这是最容易被忽略、但最重要的安全步骤：如果你的机械臂存在一个死区（某个关节物理上绝对不能转到的角度范围——比如会撞到底座或撞到另一根连杆），一定要先把这个安全范围配置好，再去跑任何真正会让机械臂动起来的工具（第5步开始的所有工具都会让它动）。**

### 为什么需要这一步

现在的IK（逆运动学）计算只检查"这个目标点在连杆长度允许的几何范围内吗"，它完全不知道你的机械结构存在死区这回事。而且新的运动控制器为了让动作丝滑，跑的速度是接近舵机最大转速的——如果方向或者零点偏移还没在这台新装好的机器上验证过，随便按一下方向键就有可能全速冲向错误的方向。这一步就是专门堵上这个风险的。

### 具体怎么做

第一步，先确认设备已经接好、通电、`calib.json`里的`hardware.servo_port`和`hardware.joint_ids`已经配置正确（可以先跑一遍`python3 main.py test-servo`确认能正常通讯，见第5步）。

然后运行：

```bash
python3 main.py set-joint-limits
```

接下来命令行会依次带你走完两个关节，每个关节的流程完全一样：

1. **屏幕会提示**类似这样的内容：
   ```
   --- joint1: torque OFF, move it by hand now (Ctrl+C when done) ---
   ```
   这表示这个关节的**舵机力矩已经被关闭**——现在可以用手自由转动它了，不会有电机的阻力，也不用担心费力气掰坏什么（这个机械臂是在水平面内运动的，不需要克服重力，所以关掉力矩之后关节不会自己往下掉，可以放心地慢慢探索）。

2. **用手转动这个关节**，尽量转遍它完整的安全活动范围——从一侧转到另一侧，中间可以来回多转几次，确保把整个范围都探索到。转动的过程中，屏幕会实时刷新一行类似这样的内容：
   ```
   current: 145.3 deg   range seen so far: [ 68.0, 180.0]
   ```
   `current`是当前角度，`range seen so far`是**从你开始转这个关节以来，观察到的最小值和最大值**——这个范围会随着你转动的角度不断扩大，不用你自己去判断"这算不算边界"，代码会自动帮你记录。

   **重要**：转动的时候，请在真正的死区/障碍物前面留一点余量，不要非要顶到最边缘才停手——比如你知道再转5°就会撞到，那就转到还剩5-10°的地方就够了，宁可保守一点。这一步记录的是"你实际探索过的范围"，不会自动帮你收缩安全边距。

3. **转够了之后，按`Ctrl+C`**。这个关节的探索就结束了，屏幕会打印出最终确定的范围，力矩也会自动重新打开。然后会自动开始下一个关节的同样流程。

4. **两个关节都测完后**，命令行会打印一个汇总，类似：
   ```
   measured limits: {'joint1': [68.0, 180.0], 'joint2': [39.5, 275.2]}
   write these to the servo's hardware registers AND calib.json? [y/N]
   ```
   输入`y`并回车，才会真正把这个范围**同时**写入两个地方：
   - **舵机自己的硬件寄存器**（这是最保底的一层——一旦写入，舵机固件本身就会拒绝转过这个边界，不管上层软件发了什么指令，哪怕软件哪里有bug也没用）
   - **calib.json里的`joint_limits_deg`**（这是软件层面的第二层保护，`main.py jog`、`manual_test/run.py`、`manual_test/gui.py`这些工具在计算目标点的时候都会检查这个范围，超出就直接拒绝、不发送任何指令，并且能给出比"硬件直接不动"更清楚的提示）

   如果输入`y`以外的任何内容（比如直接回车，或者输入`n`），就什么都不会写入，等于这次白测了，可以放心重新跑一遍这个命令。

### 如果远端臂(joint2)在近端臂(joint1)某些位置下安全范围会缩小（联动/相对死区）

这是专门用来处理**联动死区**的：有些2连杆机械臂存在这种情况——joint2的安全范围会**连续地**随joint1当前的位置变化（比如joint2跟某个固定障碍物之间的间隙，会随着joint1转动平滑地变大变小，不是"joint1在这个区域就撞、在那个区域就不撞"这种非黑即白的关系）。之前试过两种自动化的采集方式（沿边缘描线猜哪一侧受限；扫满内部按joint1分箱取min/max），实测都不靠谱。现在改成**完全手动**：用手围着安全区域的整个边界画一圈闭合轮廓，画出来的轨迹本身就直接是边界，不做任何分箱/推导/平滑处理。

**这一步的原理需要你先知道**：STS3215舵机自己的硬件限位寄存器是每个舵机各自独立的，一个舵机的寄存器没办法表达"我的限位取决于另一个舵机现在在哪"这种关系——所以联动死区**只能靠软件保护，没有硬件寄存器这层兜底**。如果这个联动碰撞的风险比较严重（不只是"摆出来的姿势难看"，而是真的会撞坏东西），强烈建议**同时加装一个物理机械挡块**作为真正的兜底，不要只依赖这里的软件配置。

采集的方法有两种，效果一样，按你手头有没有显示器来选：

#### 推荐：有显示器的话，用实时可视化工具

```bash
python3 manual_test/trace_boundary_gui.py
```

会打开一个pygame窗口。按`b`开始录制，然后用手扶着末端（两个关节的力矩都已释放），围着安全区域的**整个边界**走一圈闭合轮廓——一笔连续的动作，回到大致的起点附近。窗口会实时把走过的轨迹（换算成真实工作空间毫米坐标）连线画出来。再按一次`b`停止录制。按键：

- `b`：开始/停止录制边界轮廓（重新开始录制时会清空之前的轨迹）
- `c`：清空轨迹、停止录制
- `s` 或回车：保存到calib.json，**同时把当前窗口截图存成`joint_limits_trace.png`**（存在当前目录下，就是把屏幕上画的内容原样截下来，不是另外生成的示意图，所以看到的和存的完全一致）
- `r`：**回放**已保存的边界（见下）
- `q` / ESC：退出但不保存

保存之后可以按`r`**回放**这条边界：扭矩会重新使能，工具会真的驱动机械臂沿着整条保存好的轮廓走一圈再回到起点，这样你能同时看到仿真臂和真实机械臂沿着你刚保存的边界移动，直观确认保存的数据对不对。这个操作是让电机精确走到"死区边缘"这条线上，没有额外向内收缩的安全边距——所以回放的时候要盯紧点，觉得不对随时按`r`（或`q`/ESC）中止。

这个工具需要calib.json里已经有`joint_limits_deg`（也就是前面两个关节各自独立范围那一步必须先做完）——如果还没做，先照上面的步骤跑一遍`main.py set-joint-limits`（那一步的"联动死区"部分可以先跳过，回头用这个工具单独补）。

这个窗口同时也是检查运动学参数对不对的直观工具：画出来的机械臂就是calib.json当前L1/L2/偏移量算出来的样子——如果跟实际机械臂的姿态明显对不上（比如实际L1、L2两连杆夹角是90度，画出来却不是——如果你还没跑过`main.py calibrate`，这是正常现象），可以把机械臂手动折到一个已知的夹角（用直角尺或量角器在肘部量，默认参考值是90度，想改用`--elbow-ref-deg`指定），然后按`k`键，工具会直接从这一个姿态反解出`servo2_offset_deg`，不需要摄像头。这个方法**只能修正servo2_offset_deg（肘部的相对夹角）**，求不出`servo1_offset_deg`（需要一个绝对方向的参考）或者`elbow_offset_mm`（需要直接的物理测量——为什么视觉标定也测不出这个值，见README.md的设计说明）。单纯一个角度读数分不清肘部是往哪个转动方向折的——如果按`k`之后画面看起来更不对/像镜像了，换成`shift+k`试试。

#### 没有显示器/远程SSH：终端流程

两个关节各自的独立范围测完、打印出汇总之后，`main.py set-joint-limits`命令会接着问你：

```
trace a closed loop around the safe region's boundary? [y/N]
```

输入`y`之后：

1. **两个关节的力矩会同时关闭**。用手扶着末端，围着安全区域的整个边界走一圈闭合轮廓（跟上面GUI工具的动作完全一样），大致回到起点附近。这次调用本身就是"开始"，按`Ctrl+C`就是"结束"。

2. 走的过程中，屏幕会实时刷新已经记录的样本数和当前角度，比如：
   ```
   joint1= 152.0  joint2= 184.4   (52 samples)
   ```
   走完之后按`Ctrl+C`结束。

3. 记录到的这条轨迹**原样**存成边界——不做任何分箱/推导/平滑，你画的就是最终结果。

4. 然后会**自动生成一张示意图**（`joint_limits_trace.png`），把轨迹通过运动学换算成真实的工作空间坐标（毫米）画出来（用的是calib.json里当前的连杆长度L1/L2和底座位置，不需要你额外提供）。打开这张图，直观确认一下形状是否跟机械臂实际的死区大致吻合——这是一张辅助确认用的示意图，不是精确测量图，大致对得上就行。

最后统一确认写入时，会**先把所有内容存进calib.json，再去写舵机的硬件寄存器**——这个顺序是特意这样安排的：万一写硬件寄存器那一步遇到通讯问题失败了，你辛苦手动测出来的数据不会因此丢失，已经存好了，硬件那部分失败只会打印一条警告，不会中断整个流程。这条闭合边界**只会写进calib.json**，不会（也没办法）写进任何舵机的硬件寄存器——命令会打印一行提示提醒你这一点（`trace_boundary_gui.py`保存时也是同样的道理，只更新calib.json，不碰硬件寄存器）。

**如果轨迹上某个顶点的joint1/joint2数值，比之前两个关节各自独立测出来的范围还要略宽一点**——这是正常的，不是数据错误。徒手扫动本来就没法做到两次分毫不差。工具会自动把**两个关节**各自的独立范围都扩大成"它自己+轨迹所有顶点"的并集，再写入硬件寄存器——因为硬件寄存器只能设一个固定范围（不知道另一个关节在哪），必须覆盖你实际确认过安全的所有位置，否则会错误地拒绝掉一些刚刚确认过是安全的角度。

判断一个姿态是否"在安全范围内"，用的是**点在多边形内部**的判定（而不是"按joint1查joint2区间"），所以轨迹形状不管多不规则、甚至有凹陷都能正确处理——这也是放弃自动推导、改用完整闭合轮廓的意义所在：数据结构上就能表达任意形状，不需要凑合。

### 如果看到警告"范围超过300度"

如果某个关节测出来的范围特别宽（比如接近360°），命令会提示一个警告。这通常意味着两种情况之一：
- 你探索的时候不小心转到了角度0°/360°的接缝附近，导致记录到了错误的极值；
- 或者你的死区本身正好横跨了这个接缝——这种情况下"安全范围"反而是不连续的两段，而这套简单的min/max限位机制处理不了这种情况。

如果是第二种情况，比较好的解决办法是重新调整舵机的安装角度（转子跟连杆的相对安装位置），让**死区**（不能去的地方）卡在0°/360°接缝上，这样剩下的**安全范围**就会是一段连续的弧，这套限位机制才能正确表示。

### 怎么验证配置生效了

配置完之后，可以：
- 再跑一遍`python3 main.py set-joint-limits`，看这次汇总打印的范围是不是跟你写入的一致；
- 或者直接跑`python3 main.py jog`，试着让某个关节转到你刚配置的范围之外的角度，应该会看到类似`REJECTED: ... outside the configured joint_limits_deg`的提示，而不是真的转过去。如果配了联动边界，也可以试着指令到你确认画在轮廓外面的一个(joint1,joint2)组合，验证同样会被拒绝。

## 5. 单独测试各个部分

**完全不需要硬件**（纯逻辑测试）：
```bash
cd new && pytest tests/
```

**相机+标签检测**（不需要舵机）：
```bash
python3 main.py test-camera --watch
```
会在终端实时打印检测到的标签，Ctrl+C停止，并保存一张标注过的截图（`tag_detect.jpg`）方便你肉眼确认标签ID和位置对不对。如果相机本身打不开，先确认`libcamera-hello`能不能正常工作——这样能区分是硬件/驱动问题还是Python代码问题。

**舵机**（不需要相机；此时应该已经在第4步配置好了安全限位）：
```bash
python3 main.py test-servo
```
会ping两个关节、读取它们真实的编码器角度、把joint1轻推+5°并确认它稳定到位。默认使用`calib.json`里的`hardware.servo_port`；如果想临时用别的端口，可以加`--port /dev/ttyXXXX`。

**手动点动+扫描，跳过相机/标定**（curses终端版，或者带real-vs-target可视化的pygame版）：
```bash
python3 manual_test/run.py     # 终端版
python3 manual_test/gui.py     # 可视化版（需要能显示图形界面）
```
方向键点动末端（不按键什么都不会动）——如果你把扫描区域拖转过角度（第7步），方向键是沿着这个矩形自己倾斜的边走的，不是世界坐标的绝对x/y，所以按"上"始终是沿着你眼前这个框的一条边移动，`[`/`]`调整步长，`h`回到扫描区域中心，`t`跑一次蛇形扫描（`t`扫描过程中按`h`会中断扫描直接回中心，按`q`原地中止），`q`/ESC退出。两个工具读的是同一份`calib.json`——如果想调整扫描密度/速度，改`calib.json`的`motion`那部分就行，不用改两份重复的硬编码常量。

**原始角度交互点动**（用于底层调试）：
```bash
python3 main.py jog
```
`1 <角度>` / `2 <角度>`把joint1/joint2移动到一个绝对角度（走的是同一套运动规划器，一样是平滑的），`w`显示真实角度vs当前指令角度，`q`退出。

## 6. 完整标定流程

相机和舵机都分别确认没问题之后：

1. 把4张角标签贴在工作纸面的四个角上，末端标签要跟纸面高度基本齐平（用一个小的下垂支架就行——保证它和角标签在同一个平面上，避免相机哪怕有一点倾斜也会引入视差误差）。
2. ```bash
   python3 main.py homography
   ```
   检测四个角标签，拟合像素↔毫米的映射关系，保存。
3. ```bash
   python3 main.py calibrate
   ```
   自动遍历一组舵机角度（同样是平滑运动），读取真实编码器角度，同时观察末端标签，拟合出L1/L2/底座位置/舵机偏移。会打印每个点的误差报告，问你要不要保存（RMS<1mm：很好，1-3mm：可用，>3mm：先检查标签/安装是否有问题）。
4. ```bash
   python3 main.py selfcheck
   ```
   开机健康检查：重新核对一次标定，再做几个抽检动作。这个应该配置成每次开机自动运行一次（比如通过systemd）。

所有标定状态都存在`calib.json`里（已加入gitignore——这是每台设备各自的状态，不是源码，不需要提交；`calib.example.json`展示了完整的schema供参考）。

## 7. 把点动/扫描区域调整到真正可达的范围内

`manual_test/gui.py`/`run.py`里点动/扫描的矩形，以前直接用的是AprilTag标定工作表本身的尺寸（`workspace.width_mm`/`height_mm`）——但这个表格的四个角是你实际贴标签的物理位置，不保证跟机械臂真正能安全到达的区域（第4步测出来的`joint_limits_deg`）重合。如果点动/扫描的时候发现矩形有些角落到不了，可以配置一个更小、可以整体挪动的子矩形：

```bash
python3 manual_test/scan_area_gui.py
```

会打开一个pygame窗口，把每个可达+安全的工作空间点都涂成绿色（用的是`main.py jog`/扫描已经在用的同一套检查——IK可达性+两个关节各自的独立范围+联动死区闭合多边形，三层一起判定），同时画出标定表本身的轮廓作参考（固定，这里不能拖——要改这个得物理挪动AprilTag、重新跑`main.py homography`），并且叠加显示当前的点动/扫描矩形（没配置过的话默认是整张表）——**这个矩形可以倾斜**，斜着放往往比轴对齐更能贴合不规则形状的可达区域：

- 拖**角上的黄色手柄**：调整大小（以当前中心为基准对称缩放）
- 拖**顶边上方的紫色手柄**：绕中心旋转
- 拖**矩形内部**（不在手柄上）：整体挪动

矩形边框全部落在可达区域内时变绿，有部分不可达时变红。`f`重置成整张表（不旋转），`s`/回车保存到calib.json、**同时把当前窗口截图存成`scan_area.png`**（存在当前目录下，就是把屏幕内容原样截下来，看到的和存的完全一致，跟`trace_boundary_gui.py`的截图是同一个道理），`q`/ESC不保存退出。这个工具完全不碰扭矩、不会让机械臂动——只读编码器角度画出当前姿态作参考，所以随时都能安全运行；如果想手动把机械臂摆到某个具体位置跟图上核对，自己先用`trace_boundary_gui.py`之类的工具把扭矩松开就行。

这个工具**不会**修改`workspace.width_mm`/`height_mm`/`corner_world_mm`（标定表本身的物理尺寸，这个工具改不了也不会去改）；它保存的是一个独立的子矩形（`motion.scan_center_x_mm`/`scan_center_y_mm`/`scan_width_mm`/`scan_height_mm`/`scan_rotation_deg`），`generate_scan_path()`会优先用这个，没配置过就照旧退回整张标定表。
